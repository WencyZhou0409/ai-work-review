# AI 工作复盘助手 — 开发指南

## 理念

### 核心信念

-   **增量进步优于一步到位** - 优先选择能够编译并顺利通过测试的小步迭代。
-   **从现有代码中学习** - 在动手实现新功能前，充分研究和规划。
-   **务实优于教条** - 灵活适应项目的实际情况，而非僵守原则。
-   **清晰意图优于巧妙代码** - 追求代码的直白与易懂，避免炫技。

### 简单意味着

-   每个函数或类只承担单一职责。
-   避免进行不成熟的抽象设计。
-   拒绝使用花哨的技巧，选择最稳妥、最直接的解决方案。
-   如果一段代码需要额外的解释才能被理解，那么它本身就过于复杂了。

## 流程

### 1. 规划与分阶段

将复杂的任务拆解为 3 到 5 个明确的阶段，并在 `IMPLEMENTATION_PLAN.md` 文件中详细记录：

```markdown
## 阶段 N: [阶段名称]
**目标**: [明确的可交付成果]
**成功标准**: [可被量化和测试的成果]
**测试**: [具体的测试用例]
**状态**: [未开始 | 进行中 | 已完成]
```

-   在开发过程中，实时更新每个阶段的状态。
-   当所有阶段都完成后，删除此计划文件。

### 2. 实现流程

1.  **理解** - 深入研究代码库中已有的设计模式和实践。
2.  **测试** - 遵循测试驱动开发（TDD），首先编写失败的测试（红灯）。
3.  **实现** - 编写最精简的代码，使其能通过测试（绿灯）。
4.  **重构** - 在确保所有测试都能通过的前提下，优化和清理代码。
5.  **提交** - 撰写清晰的提交信息，并关联到相应的开发计划。

### 3. 遇到困难时（最多尝试 3 次）

如果同一问题尝试 3 次仍未解决，停下来记录问题的上下文，并寻求帮助或换一条路径。

## 规范补充

### 环境变量管理

-   所有 API Key、数据库连接字符串必须存放在各子目录的 `.env` 文件中。
-   `.env` 和 `.env.local` 必须加入 `.gitignore`，严禁提交到仓库。
-   每个子项目需提供 `.env.example` 作为配置模板。

### API 设计规范

-   后端统一返回格式：`{ "code": 0, "data": ..., "message": "ok" }`。
-   所有错误必须返回明确的 HTTP 状态码和可读的错误信息。
-   LLM 调用失败时，必须向前端返回友好的"AI 处理超时，请重试"提示。
-   API 路径使用 RESTful 风格，资源名使用复数（如 `/projects`、`/fragments`）。

### 代码提交规范

-   每次 commit 只包含一个逻辑变更。
-   commit message 格式：`<type>: <简短描述>`。
    -   常用 type：`feat`（新功能）、`fix`（修复）、`docs`（文档）、`refactor`（重构）、`test`（测试）。
-   若最近 5 条 commit 为中文，则继续使用中文提交，并在提交前向用户确认。
-   严禁提交 `.env`、本地缓存文件或未经检查的自动生成的代码。

### 跨对话记忆机制

-   在根目录维护 `PROJECT_CONTEXT.md`，记录：
    -   当前整体进度（处于哪个阶段）
    -   最近一次对话完成的核心工作
    -   当前阻塞点或待确认问题
    -   下次对话建议的切入点
-   **每次对话结束时，更新该文件。**
-   如果对话意外中断，用户在新的对话窗口中可以直接提供 `PROJECT_CONTEXT.md` 和 `PRD.md` 的内容，快速恢复上下文。

## 技术栈速查

| 模块 | 技术 | 备注 |
|------|------|------|
| Web 前端 | Next.js 14 + React + TypeScript + Tailwind CSS | 运行在 `localhost:3000` |
| 浏览器插件 | Plasmo + React + TypeScript | 最小化悬浮窗设计 |
| 后端 | Python 3.11+ + FastAPI | 运行在 `localhost:8000` |
| 数据库 | PostgreSQL / Supabase | 通过 `asyncpg` 异步连接 |
| AI 接口 | Kimi API (Moonshot AI) | 封装在 `backend/app/services/llm.py` |
