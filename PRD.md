【PRD】工作复盘AI产品
1. 概述
产品名称： AI 工作复盘助手
产品定位： 一款专为互联网从业人员（第一版主要针对数据与策略导向产品经理）打造的“Web + 浏览器插件”复盘工具。
核心价值： 解决用户在日常高强度工作中，灵感易逝、业务上下文碎片化、以及项目复盘/撰写简历耗时费力的问题。通过浏览器插件实现“无摩擦、不打断工作流”的多模态记录，并在 Web 端赋予用户绝对的控制权，利用 AI 提炼碎片信息，一键生成逻辑严密、数据详实的“汇报模式”与遵循 STAR 法则的“简历模式”文档。本产品强调“AI 起草、用户决策”，拒绝无根据的自动编造，帮助用户重建工作秩序。

2. 目标  (SMART 原则)
在初始版本上线后的初步运营阶段，期望达成以下可量化的业务与用户体验目标：
- 活跃与使用深度 (Engagement)：留存用户平均每周通过浏览器插件主动记录并打上标签的有效工作碎片15条。
- 效率提升 (Efficiency)： 核心目标用户在进行单个项目（如月度复盘、功能迭代总结）的复盘时，通过AI一键生成标准化文档，相比纯人工梳理，平均为用户节省1h的时间。
- 功能转化率 (Conversion)： 收集的碎片库中，有60%的内容最终被用户勾选并成功转化为输出。

3. 目标受众
初始版本核心受众聚焦于注重数据分析与策略制定/功能优化的产品经理。
- 日常工作特征： 工作节奏快，日常涉及大量的SQL跑数、ABT设计和观察以及用户行为分析，业务依赖于逻辑严密的因果推导。
- 行为偏好： 偏好有秩序、结构化的生活和工作方式，喜欢做规划并追求确定性。对复盘文档的质量要求极高，要求必须有明确的业务指标和动作逻辑支撑，便于反复迭代优化。
- 核心痛点：
  - 痛点1：针对有用信息丢失和碎片化的问题
  解决方案：提供全局快捷键和多模态输入（语音/文档/文本）
  解决逻辑：例如，在跑SQL时发现的小异常，或者开会后的口头总结，能在不打断工作流的情况下通过快捷键瞬间收拢到一个标签里
  - 痛点2：整理过程低效，我们需要的是讲出整个故事
  解决方案：AI瞬间重组碎片，一键生成“汇报模式”和“STAR 简历模式”两套标准化内容
  - 痛点3：市面上的有些工具过度自动化，比如自动把过期任务标为已完成，或者按照单一的模板来复盘以至于适用范围窄
  解决方案：用户拥有复盘的掌控权，允许用户手动圈选碎片，可以确认或者微调AI提取和分析的内容
  解决逻辑：AI定位是起草，而非决策，用户用AI来提效，而不必承担AI瞎编乱造的业务风险
  - 痛点4：面对庞大任务时的焦虑和迷茫
  解决方案：在最开始就建立项目tag和其核心目标
  解决逻辑：这个前置动作帮用户锚定焦点，时刻保持对业务目标的对齐

4. 使用场景
核心场景：完成一次“xxx策略”的迭代与复盘
- 场景 1：建立秩序 
  - 作为 策略产品经理，
  - 我希望 在网页端新建一个名为xxx策略的tag，并输入目标“提升xxx效率与点击率”，
  - 以便于 告诉 AI，未来所有打上这个标签的碎片，都是为了服务于这个核心业务目的，从而提供最高权重的上下文。
- 场景 2：无感捕获 (Capture
  - 作为 策略产品经理，
  - 我希望 在看竞品网站或查阅内部 BI 数据大盘时，通过点击浏览器右上角的插件图标（或快捷键），直接呼出极简悬浮框。我快速语音录入：“今天跑数据发现，改版后xxx提了 5%，但是另一个指标负向了，策略上可能需要改进xxx逻辑”。选定标签tag后回车。
  - 以便于 我在不切换页面、不打断当前分析心流的情况下，完成一次高价值策略灵感的记录。
- 场景 3：掌控复盘 
  - 作为 策略产品经理，
  - 我希望 在周五下午准备写周报时，登录工具的 Web 主界面，筛选出该标签下本周收集的 8 条碎片。我排除了 1 条最终未采纳的废弃想法，勾选了其余 7 条，点击“提取分析”。
  - 以便于 拥有绝对的信息把控权，确保参与复盘的数据和事实都是我认可的，防止 AI 在未知数据上瞎编。
- 场景 4：一键资产化 
  - 作为 策略产品经理，
  - 我希望 确认 AI 提取的关键事实无误后，同屏能直接生成两套文案：一套是适合粘贴到周报里的“汇报模式”（有数据、有逻辑和结论）；另一套是浓缩为一个或多个item的“简历模式”（STAR 法则，例如“在xxx任务中，通过制定xxx策略，最终实现xxx提升 5%”）。
  - 以便于 我只需点击“一键复制”，就能立刻将这些打磨好的文字无缝填入我的公司周报系统或个人简历中，极大节省脑力。

5. 功能需求
本 MVP 版本的系统分为三大核心模块：浏览器插件端（轻量捕获）、Web 主平台端（深度管理与输出）以及 AI 处理引擎。
5.1 浏览器插件端
核心定位：轻量、快捷、即时反馈
- F1.1 快捷呼出： 支持通过点击浏览器扩展栏 Icon 或全局快捷键（如 Cmd/Ctrl + Shift + K）在当前任意网页上方弹出极简悬浮输入框。
- F1.2 多模态输入（轻量级）：
  - 文本输入： 提供基础的多行文本输入框。
  - 语音输入： 提供“按住说话/点击录音”按钮，调用浏览器 Web Speech API 实时转写为文本。
- F1.3 项目标签绑定： 提供一个下拉列表（支持模糊搜索），展示用户在 Web 端已创建的所有 #项目标签 供用户快速点选。必须选择一个标签后才能提交。
- F1.4 AI 实时提炼与确认：
  - 用户点击“录入”后，悬浮框展示简短的 Loading 状态（后台调用 AI 进行预处理）。
  - 处理完成后，界面展示 AI 提炼出的“核心要素”（如 1-3 条结构化短句）。
  - 用户可快速浏览，点击“确认并关闭”（数据正式入库保存），或点击“修改”对 AI 提取的内容进行微调。
5.2 Web 主平台端
核心定位：项目管理、深度解析、复盘转化
- F2.1 项目/标签管理：
  - 支持新建 #项目卡片，必须输入：标签名称（如 #首页改版）和 核心目标/背景描述（如 50 字以内的一句话，用于锚定 AI 上下文）。
  - 支持查看、编辑、归档已有项目标签。
- F2.2 重度多模态输入（文档解析）：
  - 在 Web 端项目界面内，提供文件拖拽/上传区域，支持 PDF、Word 等纯文本为主的文档上传。
  - 上传后，由系统后台异步调用 AI 进行全篇提取，并将其切分为多条“核心要素碎片”存入该标签库。
- F2.3 碎片库阅览与筛选：
  - 展示所有已捕获的碎片流（按时间倒序）。
  - 支持按 #项目标签 进行筛选。
  - 单条碎片卡片应展示：录入时间、原始素材来源（语音/文本/文档）、AI 提取的核心要素。
- F2.4 复盘与编辑：
  - 提供批量复选框（Checkboxes），用户可手动勾选本次复盘需要用到的碎片，或使用“全选”功能。
  - 提供“丢弃/忽略”功能，将无关或作废的碎片移出本次结算池。
- F2.5 一键输出：
  - 用户点击“生成复盘”后，系统调用 AI 综合处理被勾选的碎片。
  - 双视角同屏展示： 界面左右分栏或上下分屏显示：
    - 视角 A：汇报模式（结构化段落，包含背景、数据、策略、结论）。
    - 视角 B：简历模式（单行/多行精炼的 STAR 法则条目）。
  - 提供显眼的“一键复制”按钮。
5.3 AI 处理引擎 (后端逻辑)
- F3.1 预处理引擎（重质不重量）： 无论输入多长，强制 AI 按业务逻辑提取核心要素（如指标变化、核心策略、动作逻辑），单次提炼上限设为 10 条，确保因果链条不丢失。
- F3.2 终态生成引擎： 接收用户勾选的碎片集合 + 该项目的“核心目标/背景”作为上下文 Prompt，严格按照“汇报”和“简历”两种设定好的系统 Prompt 模板输出，禁止模型自行发散编造无关数据。

6. 功能需求 (简版)
系统分为：浏览器插件端（轻量捕获）、Web 主平台端（深度管理与输出）及后端 AI 处理逻辑。默认全程联网运行。
6.1 浏览器插件端
- F1.1 快捷呼出： 支持点击扩展图标呼出。
- F1.2 轻量多模态输入： 提供多行文本输入框；提供语音转文字按钮（调用浏览器原生 Speech API）。
- F1.3 标签下拉绑定： 提供已建项目的 Tag 下拉列表供点选，必填项。
- F1.4 AI 实时提炼与确认： 点击录入后，后台实时提炼，界面展示提炼出的“核心要素”，用户确认看一眼后关闭或微调（建立信任感）。
6.2 Web 主平台端 
- F2.1 项目目标管理： 支持新建/编辑标签，强制要求输入“核心目标/背景描述”定调。
- F2.2 重度多模态解析： 仅在 Web 端提供文档上传区域，支持 PDF/Word 纯文本提取切片。
- F2.3 碎片筛选池： 按时间流展示碎片，支持按 Tag 筛选。
- F2.4 人工结算干预： 提供碎片批量勾选/剔除功能。
- F2.5 双视角生成器： 基于选中碎片，触发 AI 生成并左右/上下分屏展示“汇报模式”和“简历模式”，提供“一键复制”。
6.3 后端 AI 处理引擎
- F3.1 碎片化提取 (Prompt 1)： 单条素材录入时，强制提取业务逻辑要素，剔除冗余口语。
- F3.2 复盘重组 (Prompt 2)： 将多个碎片以业务逻辑串联，严禁捏造未提供的数据。

7. 技术架构
总体思路
为确保一个人也能顺利开发并极低成本上线，技术栈如下：
- 前端与插件 (零成本托管)： 使用 React 构建 Web 界面（托管于 Vercel）；使用 Plasmo 框架复用 React 组件构建浏览器插件。
- 后端服务： 使用 Python 的 FastAPI 框架处理业务逻辑和调度大模型 API。部署于 Render 或 Railway 免费层。
- 数据库 (零成本)： 使用 Supabase（基于 PostgreSQL），提供免费的云端数据存储。
- AI 接口 (按需微量计费)： 接入高性价比且逻辑能力强的大模型 API
7.1 前端开发 
- Web 端框架： 推荐 Next.js (React)。
  - 理由： 内置路由和全栈能力（API Routes），方便快速构建 Web 界面，且对后续 SEO 友好。
- 浏览器插件： 使用 Plasmo 框架 或 Manifest V3 原生开发。
  - 理由： Plasmo 是目前最流行的插件开发框架，支持 React，能让你像写 Web 网页一样写插件，大大降低 Web 端与插件端的代码复用难度。
- UI 组件库： Tailwind CSS + shadcn/ui。
  - 理由： 极致的开发效率，适合快速构建专业感强的 AI 产品界面。
7.2 后端服务 
- 核心逻辑层： FastAPI (Python)。
  - 理由： 考虑到你熟悉 Python 且后续涉及大量的 AI 处理逻辑（LLM 操作、数据清洗），FastAPI 是目前性能最强、与 AI 库集成最丝滑的轻量级框架。
- 数据库： PostgreSQL。
  - 理由： 结构化数据存储的首选。未来如果引入“向量检索（RAG）”，可以通过 pgvector 插件无缝扩展。
- AI 引擎：根据情况调用合适的API。
7.3 数据流设计
1. 捕获阶段： 插件捕获文本/语音 -> 后端 FastAPI 接收 -> 调用 LLM 进行提炼 -> 存储提炼后的“碎片”至 PostgreSQL。
2. 复盘阶段： Web 端前端请求指定 Tag 的碎片列表 -> 用户选择 -> 后端综合选中的内容 + 项目上下文 -> 调用 LLM 生成终态文档。

8. 非功能需求
- 网络依赖： 强依赖网络，无离线缓存。断网状态下界面需给出明确的“请检查网络连接”提示，禁止提交操作。
- 响应性能： 插件端调起速度需在 500ms 内；单次文本碎片的 AI 预处理确认需在 3-5 秒内返回结果。
- 数据安全： 用户数据仅对自己可见，API 调用时不应使用数据训练模型（在模型服务商处关闭数据回馈选项）。

9. 未来规划 (V2.0 备选功能)
- 数据打通： 探索一键推送生成的复盘文档至 Notion、飞书文档或企业微信工作台。
- 图表解析： 在 Web 端引入多模态视觉模型，支持上传含有数据趋势图的截图并自动提取结论。
- 智能提示： 用户很可能会遗漏某些关键信息（比如，他在插件里记录了“上线了新策略”，但忘了记录“提升了多少 DAU”，所以未来可以使复盘结果的结尾新增一个[待补充] 提示框，指出当前复盘缺失关键的量化结果数据来反向倒逼用户思考。

10. UI/UX 设计
10.1 总体设计原则
- 内容即界面： 界面尽量干净，让用户的视线 100% 聚焦在业务文本和数据上。
- 零负担 ： 插件端的操作路径必须极短，文案要清晰明白。
- 透明与掌控： AI 介入的环节必须有明确的感知（如 Loading 动画），且 AI 的产出必须支持二次确认或修改，严禁“替用户做主”的操作。
10.2 浏览器插件端规范 (无感录入)
2.1 容器与布局
- 尺寸限制： 采用紧凑型悬浮窗（建议最大宽度 360px，高度自适应但不得超过屏幕高度的一半），确保不遮挡用户正在浏览的主体网页。
- 位置： 默认在浏览器右上角扩展程序下方弹出。
2.2 核心元素与交互状态
- 项目标签区：
  - 采用带搜索功能的下拉选择器。
  - 交互细节： 默认自动选中用户上一次录入时使用的标签，减少重复点击。
- 文本/语音输入区：
  - 默认聚焦： 插件呼出时，光标必须自动聚焦在多行文本框内，支持直接 Ctrl+V 粘贴。
  - 语音入口： 文本框右下角常驻“麦克风”图标按钮。点击后变为录音状态（红色波纹动画），再次点击结束并自动将语音转为文字填入框内。
- 提炼与确认流程：
  - 录入状态：输入框激活，底部主按钮显示为 🚀 智能提取。
  - 处理状态：点击提交后，按钮变为 处理中...（附带微弱呼吸灯或旋转动画），避免用户重复点击。
  - 确认状态：展示 AI 提取出的 1-3 行核心要素，原文本框折叠或隐藏。底部展示两个等权重按钮：取消/修改 与 ✅ 确认入库。点击入库后，插件窗口自动关闭。
10.3 Web 主平台端规范
3.1 容器与布局
- 全局布局： 采用“左窄右宽”双栏布局（类似 Notion 或飞书文档）。
  - 左侧边栏 (20% 宽度)：显示项目/标签列表，提供新建按钮。
  - 右侧工作区 (80% 宽度)：动态展示所选标签下的碎片流和复盘生成区。
3.2 碎片工作台 (右侧上半部分)
- 碎片卡片流： 采用 Feed 流形式自上而下排列。单条卡片需包含：
  - 来源 Icon（标识是文本、语音还是文档提取的）。
  - 相对时间戳（如：2 小时前、昨天 15:30）。
  - 核心要素文本（AI 提炼结果）。
- 勾选交互： 卡片左侧配备显眼的 Checkbox。勾选时，卡片背景色应有轻微变化（如浅灰色变为极浅的品牌主题色），强化“已选中”的视觉反馈。
- 悬浮操作： 鼠标悬浮在卡片上时，显示 🗑️ 移除/作废 按钮。
3.3 资产输出区 (右侧下半部分)
- 无遮挡展示： 点击 生成复盘 后，输出区在碎片列表下方平滑展开（或在右侧滑出抽屉），绝对不能覆盖原始选中的碎片卡片，方便用户上下/左右对照原意。
- 双模态清晰划分： “汇报模式”与“简历模式”分为两个独立的视觉区块（Card），每个区块右上角配备全局唯一的 📋 一键复制 按钮（点击后提示“已复制到剪贴板”）。
10.4 视觉与色彩基础
- 色彩体系：
  - 背景色： 纯白 (#FFFFFF) 或极浅的灰底 (#F9FAFB)，营造专注感。
  - 品牌/主行动色： 建议采用沉稳的深蓝色或克制的紫罗兰色，传达“理性、秩序、效率”的产品性格，避免使用过于跳脱的亮色。
  - 文本色： 主文本采用深灰黑，次要信息（如时间戳）采用中灰色。
- 排版： 采用系统默认无衬线字体（如 macOS 的 San Francisco，Windows 的 Microsoft YaHei），保证跨平台阅读的清晰度和加载速度。
- 空状态： 在没有任何碎片或标签时，提供一张简洁的插画和一句引导文案（例如：“你的工作台很干净。尝试点击右上角的插件，捕捉你的第一个工作碎片吧。”），并附带快捷操作指引。

当前碎片所属的项目标签：[{项目名称}]

11.2 核心PROMPT1：Web端复盘输出
Role
你是一位互联网领域的产品经理，擅长数据分析和逻辑链条梳理。
你的任务是将用户提供的一系列离散的“工作碎片事实”，重组为两套具备极强业务穿透力的标准化文档：一套用于工作汇报，一套用于简历条目。

项目名称：[{项目名称}]

Rules
1. 事实围墙：仅能使用上方【碎片列表】中出现的信息。严禁脑补、虚构任何未提供的指标、动作和成果。严禁进行跨碎片的虚假因果连结（除非碎片逻辑上明显构成因果）。
2. 反讨好机制：禁止使用任何主观形容词（如“卓越的”、“显著的”）、奉承或价值回声。所有输出必须像技术文档一样冷静、精确。

- 禁止使用第一人称。字数控制在单条 80-120 字。
